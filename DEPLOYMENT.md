# Production deployment — Coolify + parkquest.club

This guide covers deploying ParkQuest to a Hetzner VPS through Coolify for
the beta launch at **parkquest.club**.

## Architecture

```
Internet -> Coolify Proxy (Caddy/SSL) -> App (port 3000)
                                          |
                                          v
                              Coolify PostgreSQL (internal URL)
```

- The Next.js app runs in a Coolify-managed container (port 3000).
- PostgreSQL runs as a separate Coolify database resource, **not inside the app
  container**.
- The database is **private/internal** to the Coolify project — no public port.
- Coolify's Caddy proxy handles HTTPS termination.

---

## Prerequisites

- A Hetzner VPS (4 GB RAM recommended; see Resource check below)
- Coolify installed on that VPS
- The domain `parkquest.club` pointed to the VPS IP
- GitHub repository access for the deploy key

## Step-by-step checklist

### 1. Create a Coolify Project

1. Open your Coolify dashboard.
2. Click **Projects > New project**.
3. Name it `ParkQuest`.

### 2. Add a PostgreSQL database resource

1. Inside the project, click **Databases > New database**.
2. Choose **PostgreSQL**.
3. Settings:
   - **Name:** `parkquest-db`
   - **Postgres version:** 16
   - **Credentials:** auto-generated (or set a strong password)
   - **Persistent storage:** enabled (Coolify manages this)
   - **Backups:** enable **before** beta users join
   - **Port:** leave empty (do not expose publicly)
4. After creation, copy the **Internal Database URL**. It looks like:
   ```
   postgresql://user:password@generated-hostname:5432/parkquest
   ```
   Paste this URL for the `DATABASE_URL` env var. The hostname is generated by
   Coolify and resolves within the project network. It may differ from the
   example above.

### 3. Add the application from GitHub

1. Click **Applications > New application**.
2. Choose **Public GitHub repository**.
3. Enter: `https://github.com/BDubDesigns/parkquest`
4. **Branch:** `main`
5. **Build pack:** **Dockerfile** (select this explicitly — Nixpacks may not use
   the provided Dockerfile correctly for Next.js standalone output).
6. **Port:** `3000`
7. **Health check path:** `/api/health`

### 4. Configure environment variables

Add these under the application's **Environment variables**:

| Variable             | Value                            | Notes                                               |
| -------------------- | -------------------------------- | --------------------------------------------------- |
| `DATABASE_URL`       | `postgresql://...`               | Use the Coolify PostgreSQL internal URL from step 2 |
| `BETTER_AUTH_SECRET` | `openssl rand -base64 48` output | Required by Better Auth                             |
| `BETTER_AUTH_URL`    | `https://parkquest.club`         | Must match the production domain exactly            |
| `NODE_ENV`           | `production`                     | Next.js uses this automatically                     |
| `HOSTNAME`           | `0.0.0.0`                        | Binds to all interfaces inside the container        |
| `PORT`               | `3000`                           | Must match the app listen port                      |

> **Never** commit `.env` — the `.gitignore` already excludes it. Use Coolify's
> environment variable UI instead.

### 5. Run database migrations

The production Docker image includes migration tooling under `/app/tools/`.
After the first deployment (the app may fail because no tables exist yet),
run migrations from the Coolify terminal:

**Option A — Coolify "Execute command":**

```bash
cd /app/tools && npm run db:migrate
```

**Option B — via docker exec (from the VPS host):**

```bash
docker exec <container-name> sh -c "cd /app/tools && npm run db:migrate"
```

### 6. Seed public park data

```bash
# From Coolify "Execute command":
cd /app/tools && ALLOW_PRODUCTION_SEED=1 npm run db:seed
```

This inserts:

- 1 region (`Bellingham, WA`)
- 17 amenities
- 3 sticker/badge definitions
- 4 daily quest definitions
- Bellingham park records with verified amenity links

> **Important:** The seed script creates only public atlas data — no test users
> or test visits. It requires `ALLOW_PRODUCTION_SEED=1` when `NODE_ENV` is
> `production` as a safety check. It is safe to run once on a fresh production
> DB, and harmless to re-run (uses `onConflictDoNothing` / `onConflictDoUpdate`).

### 7. Connect the domain

1. In the application settings, set **Domain:** `parkquest.club`.
2. Coolify will automatically provision a Let's Encrypt SSL certificate via Caddy.
3. Wait for the SSL status to show **active**.
4. Test: visit `https://parkquest.club/api/health` — should return `{"status":"ok"}`.

### 8. Enable backups

Before real users are on-boarded:

1. Go to **Databases > parkquest-db > Backups**.
2. Enable scheduled backups.
3. Choose a retention policy (daily backups, keep 7 days is a reasonable start).

## Production resource check

Run these on the Hetzner VPS to verify adequate resources:

```bash
# Memory
free -h

# Disk
df -h

# Container resource usage
docker stats
```

Watch during build and after smoke test. Scale recommendations:

| Resource | Minimum | Recommended | Notes                                |
| -------- | ------- | ----------- | ------------------------------------ |
| RAM      | 2 GB    | 4 GB        | Next.js build is memory-intensive    |
| Disk     | 20 GB   | 40 GB       | Docker images, DB, backups add up    |
| Swap     | 2 GB    | 2 GB        | Helps on small VPS during peak build |

**When to upgrade** to a larger VPS:

- Builds consistently fail with out-of-memory errors.
- Memory pressure is high during normal operation (`free -h` shows <500 MB free).
- Disk usage exceeds 80%.
- Runtime container CPU or memory usage is unstable (`docker stats`).

## Environment variables reference

### `DATABASE_URL` (required)

PostgreSQL connection string. In production, use the Coolify PostgreSQL internal URL:

```
postgresql://user:password@coolify-generated-hostname:5432/parkquest
```

Copy the exact URL from the Coolify database resource page. The hostname is
generated by Coolify and resolves within the project network.

### `BETTER_AUTH_SECRET` (required)

Generate with:

```bash
openssl rand -base64 48
```

This secret is used to sign auth session cookies. Changing it invalidates all
active sessions. Keep it stable across deployments.

### `BETTER_AUTH_URL` (required)

Must be the **exact canonical URL** of the production app:

```
https://parkquest.club
```

Better Auth uses this to construct redirect URLs (sign-in, email verification,
OAuth callbacks). If this does not match the deployed domain, auth flows will
break.

### `NODE_ENV` (set by Coolify)

Automatically `production` during deployment. The `db` singleton uses this to
avoid caching the pool reference in the global scope during development:

```ts
if (process.env.NODE_ENV !== "production") globalForDb.dbPool = pool;
```

### `PORT` and `HOSTNAME`

| Variable   | Value     | Purpose                                            |
| ---------- | --------- | -------------------------------------------------- |
| `PORT`     | `3000`    | App listen port (matches Dockerfile `EXPOSE 3000`) |
| `HOSTNAME` | `0.0.0.0` | Binds to all interfaces inside the container       |

## Database guidance

- **Use Coolify-managed PostgreSQL** — create a database resource in the same
  project as the app. The internal DNS hostname resolves automatically.
- **Do not run Postgres inside the app container.** It creates version conflicts,
  complicates backups, and wastes resources.
- **Do not expose the database publicly.** Coolify PostgreSQL resources are
  internal by default. Keep it that way unless you have a specific documented
  reason to add a public port.
- **Enable persistent storage.** Coolify attaches a Docker volume to the
  PostgreSQL container. The data survives container restarts and redeployments.
- **Enable backups** before real users register.
- **Migrations** must be run once after the first deployment (the Docker image
  does not include migration tooling). Use Coolify's "Execute command" feature
  or `docker exec`. See step 5 above.
- **Seeding** public park data requires `ALLOW_PRODUCTION_SEED=1`. See step 6
  above.

## Better Auth production notes

- **`BETTER_AUTH_URL` must be `https://parkquest.club`** — not `http` and not
  `localhost`. Better Auth uses this for CSRF protection and redirect
  construction. A mismatch will cause sign-in failures.
- **Secure cookies:** In `NODE_ENV=production`, Next.js and Better Auth
  automatically set `Secure` and `SameSite` attributes on cookies (handled by
  the framework when the request is served over HTTPS). Coolify's Caddy proxy
  terminates TLS, so the app sees `x-forwarded-proto: https`.
- **Session secret:** The `BETTER_AUTH_SECRET` must be a strong random value.
  Changing it invalidates all active sessions — treat it like a database
  password.
- **No user data leak:** Better Auth schemas (`user`, `session`, `account`
  tables) are only queried server-side. No PII is exposed to the client via
  API routes unless explicitly coded.

## Privacy basics

ParkQuest is designed with family privacy as a foundation:

- **All family progress is private to that family.** Stamps, Adventure Points,
  stickers, memories, and nicknames are scoped by `familyGroupId` and never
  leak into signed-out views or other families' sessions.
- **Notes/memories are private.** The "memory" text entered during stamping is
  stored per-visit and only returned for the owning family.
- **No GPS location is collected.** ParkQuest uses Leaflet + OpenStreetMap tiles
  but does not request device location.
- **No social features.** There is no friend feed, no public profile, and no
  child account system.
- **No email marketing.** Email addresses are collected only for authentication.

For privacy requests or data deletion, email privacy@parkquest.club.
The email forwarder should be configured before beta users are invited.
